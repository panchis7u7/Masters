\section{Introduction}
\label{sec:introduction} 

As modern networks become increasingly software-defined and API-driven, automation platforms expose numerous APIs for configuration, monitoring, troubleshooting, and operational management. However, the complexity and volume of technical documentation and hardcoded API interactions make it difficult for both automation systems and network engineers to efficiently identify and utilize the appropriate APIs, limiting the effectiveness of network automation.

Retrieval-Augmented Generation (RAG) architectures provide a promising approach to this challenge by combining large language models with external knowledge sources. Through semantic retrieval, these systems can access and reason over technical documentation in real time, enabling AI agents to discover relevant APIs and support automation workflows beyond traditional keyword-based search \parencite{lewis2020rag,wu2024ragsurvey}.

Vector databases constitute a key component of RAG architectures by enabling efficient similarity searches over embedded representations of documentation. This capability may improve the discovery of relevant technical information and facilitate contextual reasoning over large API repositories \parencite{johnson2019faiss}.

Despite their growing adoption, the effectiveness of vector databases for technical documentation retrieval in network automation remains insufficiently understood. This research therefore investigates the use of vector databases for AI-driven retrieval of SDN API documentation and evaluates the performance of FAISS, PGVector, and Neo4j in terms of retrieval accuracy, response quality, and operational performance.

\section{Materials and Methods}
\label{sec:Materials_and_Methods}
\subsection{Hypothesis}

\noindent\textbf{Hypothesis 1 (Main):}
The incorporation of vector databases into RAG-based AI agents improves knowledge retrieval and API discovery in SDN documentation compared to traditional search methods.

\noindent\textbf{Hypothesis 2:}
Vector database-based retrieval achieves higher retrieval accuracy than keyword-based search.

\noindent\textbf{Hypothesis 3:}
Implementations of vector databases like FAISS, PGVector, and Neo4j exhibit statistically significant differences in retrieval accuracy, response quality, and latency.

\noindent\textbf{Null Hypothesis:}
The choice of vector database does not significantly affect the quality of responses generated by RAG-based AI agents.

Hypothesis 1 is the base of the investigation, while Hypotheses 2 and 3 are interdependant specific sub-hypotheses that will be tested to evaluate the overall impact. It might be possible that specific algorithms may align better with a specific set of data, for which hypothesis 3 is designed to evaluate the performance of different implementations, in this case: FAISS, PGVector, and Neo4j.

\subsection{Data Acquisition and Ethical Considerations}

The experimental corpus was constructed from publicly available Cisco SD-WAN API DevNet documentation \parencite{cisco_sdwan_docs} and the OpenAPI specification provided by Cisco SD-WAN vManage \parencite{cisco_vmanage_openapi}.

API documentation content is processed into embeddings and stored within vector databases solely to support retrieval experiments. The original documents remain the intellectual property of their respective owners and are not redistributed as part of this research. References to all documentation sources are provided to ensure transparency and reproducibility.

As a pre-processing task prior to vectorization, the key-value pairs are isolated and converted into natural language strings. For example, the API endpoint for retrieving device information might be transformed into a string like:

\begin{quote}
\textit{Administration - Audit Log}\\
\textit{Endpoint: GET /auditlog/page}\\
\textit{Operation: getStatBulkRawPropertyData}\\
\textit{Description: Get raw property data in bulk.}\\
\textit{Parameters:}\\
\textit{ + query (required string)}\\
\textit{ + scrollId (optional string)}\\
\textit{ + count (required integer)}\\
\textit{Returns: Audit log data.}
\end{quote}

This insures that each structured JSON entry transforms to a single string chunk capturing full context. 
\subsection{Data Treatment}

The ingestion pipeline will include text extraction, cleaning, and embedding generation using a consistent model (Qwen3-Embedding-4B, which is specifically designed for text embedding and ranking tasks) across all databases to ensure comparability. \subsection{Evaluation Metrics}
Retrieval results will be evaluated against a predefined set of queries designed to test various aspects of API discovery and information retrieval. Queries are subdivided into 3 groups and categorized based on complexity and relevance to typical network automation tasks:

\begin{itemize}
    \item \textbf{Group 1: Basic API Discovery}: Queries that require identifying specific API endpoints or parameters.
    \item \textbf{Group 2: Contextual Understanding}: Queries that require understanding the context of API usage, such as dependencies or common use cases.
    \item \textbf{Group 3: Complex Reasoning}: Queries that require multi-step reasoning or synthesis of information across multiple documentation sections.
\end{itemize}

Each group contains 15 queries, resulting in a total of 45 queries for evaluation. 

Retrieval accuracy will be evaluated on 4 different factors based on the relevance of retrieved documents to the query.

\begin{itemize}
    \item \textbf{Precision@K}: Proportion of relevant documents in top K results.
    \[
    \frac{\text{Number of relevant documents retrieved in top K}}{K}
    \]
    \item \textbf{Recall@K}: The proportion of relevant documents that were retrieved out of all relevant documents within the top K results:
    \[
    \frac{\text{Number of relevant documents retrieved in top K}}{\text{Total number of relevant documents}}
    \]
    \item \textbf{MRR (Mean Reciprocal Rank)}: The average of the reciprocal ranks of the first relevant document for each query:
    \[
    \frac{1}{|Q|} \sum_{i=1}^{|Q|} \frac{1}{\text{rank}_i}
    \]
    where \(\text{rank}_i\) is the rank of the first relevant document for query \(i\).
    \item \textbf{F1-score@K}: The harmonic mean of Precision@K and Recall@K:
    \[
    2 \cdot \frac{\text{Precision@K} \cdot \text{Recall@K}}{\text{Precision@K} + \text{Recall@K}}
    \]
\end{itemize}

Response quality will be assessed through human evaluation, where the usefulness and correctness of the retrieved information are rated on a Likert scale. 

Latency will be measured as the time taken from query submission to retrieval response, where automation will be used to ensure consistent timing across all tests.
